# Story 9.1 — Integracao Automatica com APIs de Licitacao (PNCP + Compras.gov)

**Epic:** 9 — Monitoramento Automatico de Editais
**Status:** Draft
**Prioridade:** Alta
**Estimativa:** L (Large)

---

## Objetivo

Integrar o LicitaFlow com as APIs publicas do PNCP e Compras.gov.br para buscar automaticamente editais de cesta basica / generos alimenticios, armazenar no Supabase e notificar o usuario quando houver oportunidade compativel.

---

## Contexto

Hoje o fornecedor precisa acessar manualmente os portais de licitacao todo dia. Com essa integracao, o app faz isso automaticamente 2-3x por dia, filtrando por objeto, modalidade e regiao.

### APIs Identificadas

| API | Base URL | Auth | Escopo |
|-----|----------|------|--------|
| **PNCP Consulta** | `https://pncp.gov.br/api/consulta` | Nenhuma (publica) | Federal + Estadual + Municipal |
| **Compras.gov.br** | `https://dadosabertos.compras.gov.br` | Nenhuma (publica) | Federal (historico + precos) |

### Endpoints Principais

**PNCP — Busca de Editais:**
```
GET /v1/contratacoes/publicacao
  ?dataInicial=YYYYMMDD
  &dataFinal=YYYYMMDD
  &codigoModalidadeContratacao=6  (pregao eletronico)
  &uf=SP
  &pagina=1
  &tamanhoPagina=50
```

**PNCP — Editais com Proposta Aberta:**
```
GET /v1/contratacoes/proposta
  (mesmos filtros — retorna apenas com prazo aberto)
```

**PNCP — Detalhe de Contratacao:**
```
GET /v1/orgaos/{cnpj}/compras/{ano}/{sequencial}
```

**Compras.gov — Pesquisa de Preco:**
```
GET /modulo-pesquisa-preco/1_consultarMaterial
  ?codigoItemCatalogo=...
  &estado=SP
  &dataCompraInicio=YYYY-MM-DD
  &dataCompraFim=YYYY-MM-DD
```

**Compras.gov — Historico de Pregoes:**
```
GET /modulo-legado/3_consultarPregoes
  ?dt_data_edital_inicial=YYYY-MM-DD
  &dt_data_edital_final=YYYY-MM-DD
```

### Codigos de Modalidade (PNCP)

| Codigo | Modalidade |
|--------|-----------|
| 6 | Pregao Eletronico |
| 8 | Dispensa de Licitacao |
| 4 | Concorrencia Eletronica |

## Criterios de Aceitacao

### Task 1: Schema de Editais no Supabase
- [ ] Criar migration com tabela `monitored_editals` (dados brutos da API)
- [ ] Campos: `pncp_id`, `orgao_cnpj`, `orgao_nome`, `objeto`, `valor_estimado`, `modalidade`, `uf`, `municipio`, `data_publicacao`, `data_abertura`, `link_sistema_origem`, `status_sync`, `raw_json`
- [ ] RLS: tenant_id based (cada empresa ve so os seus)
- [ ] Indice em `objeto` (full-text search) e `data_publicacao`

### Task 2: Service de Integracao PNCP
- [ ] Criar `lib/services/pncp-api.ts`
- [ ] Funcao `fetchEditaisByDate(dataInicial, dataFinal, uf, modalidade)`
- [ ] Funcao `fetchEditaisComPropostaAberta(uf, modalidade)`
- [ ] Funcao `fetchDetalheContratacao(cnpj, ano, sequencial)`
- [ ] Paginacao automatica (max 50 por pagina)
- [ ] Rate limiting: max 2 req/segundo
- [ ] Retry com backoff exponencial (3 tentativas)
- [ ] Tratamento de erros (400, 204, 500)

### Task 3: Filtro Inteligente de Relevancia
- [ ] Criar `lib/services/edital-filter.ts`
- [ ] Filtrar por palavras-chave no `objetoCompra`: "cesta basica", "generos alimenticios", "kit alimentacao", "alimentos nao pereciveis", "cesta de alimentos"
- [ ] Filtrar por UF/municipio configurado no perfil do tenant
- [ ] Filtrar por valor estimado (min/max configuravel)
- [ ] Score de relevancia (0-100) baseado em match de keywords + proximidade geografica
- [ ] Salvar apenas editais com score >= threshold configuravel

### Task 4: Cron Job de Sincronizacao
- [ ] Criar Supabase Edge Function `sync-editais`
- [ ] Executar via `pg_cron` ou via Vercel Cron: 3x/dia (08h, 12h, 18h)
- [ ] Buscar editais dos ultimos 2 dias (overlap pra nao perder nada)
- [ ] Upsert no banco (evitar duplicatas via `pncp_id`)
- [ ] Registrar log de sync: `sync_logs` (data, qtd_encontrados, qtd_relevantes, erros)
- [ ] Notificacao via tabela `notifications` quando edital relevante encontrado

### Task 5: API Route no Next.js
- [ ] `GET /api/editais/sync` — trigger manual de sincronizacao
- [ ] `GET /api/editais` — listar editais monitorados (com filtros, paginacao)
- [ ] `GET /api/editais/[id]` — detalhe de edital (busca complementar no PNCP se necessario)
- [ ] `POST /api/editais/[id]/analyze` — marcar GO/NO-GO
- [ ] Autenticacao via Supabase session (middleware ja configurado)

### Task 6: UI — Pagina de Editais Monitorados
- [ ] Pagina `/dashboard/editais` com lista de editais encontrados
- [ ] Cards com: objeto, orgao, valor estimado, UF, data abertura, score de relevancia
- [ ] Filtros: UF, modalidade, valor min/max, status (novo, analisado, GO, NO-GO)
- [ ] Badge de status: "Novo" (azul), "GO" (verde), "NO-GO" (vermelho), "Analisado" (cinza)
- [ ] Botao "Analisar" que abre o detalhe do edital
- [ ] Indicador de ultima sincronizacao
- [ ] Botao "Sincronizar agora" (trigger manual)

### Task 7: Configuracao de Monitoramento por Tenant
- [ ] Pagina `/dashboard/configuracoes/monitoramento`
- [ ] Campos: UFs de interesse (multi-select), keywords customizaveis, valor min/max, frequencia de notificacao
- [ ] Salvar em tabela `tenant_monitoring_config`
- [ ] Default: UF do endereco do tenant, keywords padrao de cesta basica

### Task 8: Integracao Compras.gov (Complementar)
- [ ] Criar `lib/services/compras-gov-api.ts`
- [ ] Funcao `fetchHistoricoPrecos(codigoItem, estado, periodo)` — pesquisa de precos praticados
- [ ] Funcao `fetchPregoes(dataInicial, dataFinal)` — pregoes legado
- [ ] Exibir historico de precos no detalhe do edital (ajuda na estrategia de lance)

## Dependencias Tecnicas

- Supabase Edge Functions (ou Vercel Cron)
- `pg_cron` extension no Supabase (se usar Edge Function agendada)
- Fetch API nativa (sem lib externa necessaria — APIs sao REST simples)

---

## Riscos e Mitigacoes

| Risco | Mitigacao |
|-------|----------|
| API do PNCP fora do ar | Retry com backoff + cache dos ultimos dados |
| Rate limit nao documentado | Limitar a 2 req/s do nosso lado |
| Mudanca na API | Versionar client, monitorar erros |
| Muito volume de dados | Filtro de relevancia agressivo, paginacao |
| Editais duplicados | Upsert via `pncp_id` unico |
